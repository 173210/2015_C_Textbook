1.C言語とC++の簡単な歴史

C言語とはUNIXというオペレーティングシステムを開発するために作られた言語だそうです。
初期はUNIXの作者であるKen Tompsonはアセンブラでプログラムを書いていたが、それが嫌になったのか「B」という言語を開発しました。
Brian Kernighan によって改良を加えられ、NewB(NB)を経てC言語へと発展していきました。こうして出来たのが今で言うK&Rです。
こうして出来たC言語はあちこちに広がりましたが、機能があちこちで独自拡張されていきました
これに収拾をつけるべくANCI(American National Standards Institute)とISO(International Organization for Standardization)は共同でC言語の規格を発表した。
これが今日C89とかANCI-C89とかISO-C90とか呼ばれるものである(それぞれ1989年と1990年に規格を発行した)。

さて、話は少し巻き戻り1979年、C with Classesと呼ばれる言語の開発が始まりました。
その名の通り、classを持つC言語で、1983年にはC++という名前に変わり、演算子と関数のオーバーロードや参照、const型、行コメント(//)などが追加されました。
1985年、『The C++ Programming Language』という事実上の標準規格Version1.0である本が発売されました。
1989年、『The Annotated C++ Reference Manual (ARM)』という事実上の標準規格Version2.0である本が発売されました。
なお、C++はC99の登場まで完全な互換性を持っていました。

C++はその元となったC言語にも大きな影響を与え、1999年、ISOとIEC(International Electrotechnical Commission)が共同で新たな規格を発行しました。
これが現在最も一般的に使われているC99と呼ばれる規格で、変数宣言位置の自由化やbool型、行コメント、inline関数などC++の機能が逆輸入されました。

同じく1998年、C++にも標準化の波が押し寄せ、C++99とよばれる規格が発行されました。この時いまやC+erの常識であるSTLと呼ばれる部分が規格に入りました。
2003年のC++03、2007年のC++TR1によるC99の機能の輸入を経てC++は発展していきました。

初めC++0xとかC++09とか言われたC++1xの規格は、策定が遅れに遅れ、2011年8月、ついにC++11とよばれる規格がサポートされました。
右辺値参照、型推論、range-base for、constexpr、lambda、enum class、nullptr、可変長引数template、Unicode文字列リテラル、ユーザー定義リテラル、マルチスレッド、正規表現、改良されたスマートポインター、新たな乱数生成など、
それまでのC++と全く別物になったと言っていいくらいの大幅な機能拡張が行われたわけです。

C言語もC11と呼ばれる新規格を発表し、セキュリティ上大問題となっていたgets関数の廃止、_sサフィックス付き関数の追加、劣化版C++のマルチスレッド機能、型判定マクロ(劣化版template)、Unicode文字列リテラルの対応などをサポートしたわけですが、C++11のインパクトが大きすぎて多くの人に忘れられがちな規格となっています。

2014年、C++14と呼ばれる新規格が発行され、2進数リテラルの対応、型推論の強化、constexpr関数の制限緩和、ジェネリックlambdaなどのC++11の小規模アップデートが行われ、
そして今C++17の策定が進んでいます。

なんかいっぱい書いたけどべつに覚える必要ないからね？ただC++11とかC99とかC++14とか言われたらなんかそんなのあったな～と思い出してくれればいいです。

2.プログラ厶言語とはなにか

wordだって、excelだって、ニコニコだって、モンハンだって、エロゲだって、みんなみんなプログラムでできている友達なんだ！

というように、コンピュータで動くありとあらゆるものはプログラムでできています。プログラムを動かすには、我々人間が機械に命令を与える必要があります。命令文をプログラム言語で書くことによって、機械に命令、実行させることができます。

プログラミング言語で書かれたプログラムを読み込んで、それを機械への命令に変換して実行するのですが、その方法は大きく分けて２通りあります。

インタープリタ方式とコンパイル方式

インタープリタ方式とは、プログラミング言語で書かれたプログラムを読み込んで、直ちにそれを機械への命令に変換して実行してくれます。ある程度まとまってれば、完成してなかったり、適当なこと書いていてもその場で実行して結果を確かめることができるので便利です。主にweb系のプログラムにこれが用いられます。現にこのテキストもhtml,javascriptという言語で書かれているので、この方法が用いられています。この方式のデメリットは、１行１行翻訳して実行してるので実行速度が落ちてしまいます。ですので、動きのある凝ったゲームとか作りたいならこの方式はあまり推奨されないでしょう。

コンパイル方式とは、プログラミング言語で書かれたプログラムを、機械が理解できるデータに変換して、普通はそれをファイルとして保存するということをします。この保存されたファイルの拡張子は.exeだったり。実行の段階では、もう構文解析とか機械語への翻訳は終わっているので、最適な実行速度が得られるわけです。まぁ、バグを取り除く作業（デバッグ作業）はいちいち大変なんですけどね汗　C言語はこの方式をとっています。

4.どんな流れでコードが処理されているか

原則として、コードは上から下にマシンは読み取ります。

例えば

int a=10;
printf("a=%d",a);

とすると

a=10

となるが

printf("a=%d",a);
int a=10;

という順番にすると、コンパイラに「printf内のaってなんすか？」って怒られます

関数を宣言するときも同じです。

6.ビルドの流れ

プリプロセッサ
プリプロセッサとは、#includeや#define,#ifndef,#endifなどのプリプロセッサ分を解釈し、コンパイラへの入力を作成します。
ざっくり言うと、#includeとか#defineが先に読み込まれるってことだね。
コンパイラ
コンパイラは、ソースコードを機械語に変換します。機械語に翻訳したあと、オブジェクトファイル（windowsなら拡張子.obj）を作ります。 もっというとアセンブリ言語に変換するのがコンパイラーで、アセンブリ言語からオブジェクトファイルを作るのはアセンブラーの仕事です。が、そこまで意識することはあまりないです。
リンカー
リンカーとは、オブジェクトファイル（複数個の場合がほとんど）とライブラリを結びつける働きをします。オブジェクトファイルにバグがあると正常に機能しません。
またC++の場合マングリングというC++のオーバーロード機能を正常に動作させるための機能がコンパイラーによって実装がまちまちなため、違うコンパイラーで作ったオブジェクトファイル、ライブラリー同士をリンクさせることは不可能に近いです。
よくソースコードにextern "C"と書いてあるのはこのためです。
この時点でクローバル変数、static変数などの配置を決めていたりしますが、それについてはこのテキストでは解説しません。
makeファイル
makeファイルには例えば「○○.cppというソースファイルをコンパイルして、○○.objを作成する」といったようにビルドすると気の手順が細かく書かれています。
https://github.com/yumetodo/2015_C_Textbook/blob/gh-pages/src/strstr/GNUmakefile
みたいなやつです。
ソースコードに変更があった時だけコンパイルとかを実行できます。別にコンパイル以外のことも実行できるんですけどね。
なお大規模になるとmakefileからmakefileを呼び出すことができます。
[Visual Studio]プロジェクトとソリューション
makeファイルは細かく色々指定できて確かに便利なんですが、いくら殆どコピペできると言ってもmakefileを書くのはめんdです。
Visual Studioにはその代わりにプロジェクト(拡張子vcxproj)とソリューション(拡張子sln)があります。
普通ソリューションには多くても3つ、殆どの場合1つのプロジェクトがあります。プロジェクトごとにいろいろオプションを変えられたりするわけです。
便利。これのお陰で「実際この記事を書いてる私はmakeファイルを作ったことないです。」という発言が飛び出るくらいには。

デバッグ、リリース

デバッグとは、デ(de:取り除く)バッグ(bug:バグを)の意味です。どの時点でバグが起きたのか、今どんな状態なのかを確認することができます。

リリースとは、リリース（release:公開）の意味です。ゲームや作品を公開するときに使います。

7.プログラミングの鉄則

多くのプログラマーの中での共通認識が有ります。それは

めんd

です。いかに面倒臭がるか、いかに手を抜くか、いかに楽してプログラミングするか、という意味です。そして僕からもう一言。

プログラム　書かなければ　バグらない

当たり前のようで忘れがちです。ついつい一から自分で作ろうとしてしまいます。あなたがやろうとしていることは大抵誰かがすでにやっています

まとめると、可能な限り書くコード量を減らすことが大事ということになります。そのために先ほど言ったライブラリーがあるのです。

8.コンパイラーの種類

組み込みまで考えだすと数えきれないほどのコンパイラーが存在しますが、有名どこを上げると概ね4つになると思います。

Intel C Compiler(icc)
おそらく最強のコンパイラー。比較的優秀なIDE(統合開発環境)と世界最強のコンパイラー。これに勝るものはなし。ただしお値段10万円ほど。VCほどじゃないけど少し規格準拠が遅い。
GNU C Compiler(gcc)
GNU Public License(GPL)で開発が続く超有名無料のコンパイラー。そこそこ規格準拠も早く、大抵のLinuxには標準で入っている。コンパイル環境を選ばない。
Clang
gccの対抗馬でgccを完全に置き換えることを目指すコンパイラー。University of Illinois/NCSA Open Source Licenseで開発されており、gccよりも規格準拠が早いことが特徴。
ただしWindowsではコマンドプロンプトがまともにUTF-8に対応しないためminttyなどの別のターミナルエミュレーターなしには使いたくない。
Visual Studio(VC)
Microsoftが開発しているIDEでコンパイラーも付属。話しにならないくらい規格準拠が遅いが、IDEのデバッグ機能が他の追従を許さないくらい優秀すぎるため、幅広く使われている。
これとMSOfficeがLinuxで使えるならWindowsなんて糞OSいらないのに。コンパイラーだけclangに差し替えるclang-clというプロジェクトもあるけど実用にはまだ耐えない。
